# 클러스터링 인덱스와 논 클러스터링 인덱스란 무엇일까요?

### Clustered Index
> 데이터 행의 원래 순서대로 저장되어 있는 인덱스를 의미합니다.

즉, 인덱스의 **리프 페이지가 실제 데이터가 저장된 페이지 그 자체**입니다.
<br>
<br>
저장 그 순서 그대로 있기 때문에 각 테이블은 단 하나의 클러스터형 인덱스를 가질 수 있습니다.

또한 행 데이터를 지정한 열에 맞추어 자동으로 정렬합니다.

이미 데이터가 모여있기 때문에 범위 검색 시 유리합니다. 

#### 예시로 살펴보자!

영어사전이 알파벳 순으로 되어 있어서 그 자체가 인덱스가 되는 경우입니다.

우리가 DB를 만들 때 하나의 PK를 잡아둘 것 입니다.

이를 지정해두면 이 PK를 기준으로 하나의 클러스터형 인덱스를 자동으로 가지게 됩니다.

## 논 클러스터링 인덱스

> 이는 Secondary Index라 불리기도 합니다.

실제 데이터의 위치와 별개로 저장되는 인덱스입니다.

즉, 실제 데이터 위치는 그대로 이지만, 이 위치만(포인터) 따로 모아서 정렬해 둔 별도의 페이지입니다.

<br>
보조형 인덱스이며, 기준이 되는 컬럼을 기준으로 실제 데이터와 다른 **별도의 순서**를 가집니다.

그렇기에 각 테이블은 여러 개의 보조 인덱스를 가질 수 있습니다.

단일 값을 찾는데는 빠르지만, 찾은 뒤 실제 데이터를 가져오는 작업이 추가로 필요합니다.

#### 예시로 살펴보자 !

책의 목차를 보면 우리는 몇 페이지에 해당 내용이 나오는 지 바로 알 수 있습니다.

그러나 우리는 그 해당 페이지로 이동해야 합니다.

<br>
<br>
우리가 DB를 만들 때 email을 `UNIQUE` 설정을 하면 그 컬럼을 기준으로 하는 하나의 보조형 인덱스가 생깁니다.

## 한 줄 정리
> 가장 큰 차이는 실제 데이터의 물리적 정렬 여부입니다.
> 클러스터링 인덱스는 인덱스의 리프 페이지가 곧 데이터 페이지 자체로, 테이블 자체가 인덱스 순서대로 정렬되어 저장됩니다. 영어 사전처럼 데이터 자체가 순서대로 모여 있어 범위 검색에 유리하며, 테이블 당 하나만 생성이 가능합니다. 주로 Primary Key가 이 역할을 수행합니다.
> 논-클러스터링 인덱스는 데이터 자체를 정렬하지 않고 별도 공간에 데이터 위치 정보를 정렬해 관리합니다.한 테이블에 여러 개를 생성할 수 있습니다.
> 즉, 클러스터링 인덱스는 데이터 저장 방식 자체를 결정하고, 논 클러스터링 인덱스는 조회를 돕는 추가 도구입니다.


### 참고 자료 및 첨언..

저는 여기서 헷갈렸던 것이

리프 페이지 == 데이터 라고 했는데, B Tree와 다르게 B+ Tree는 상위에 실제 데이터가 없고, 오직 가이드 역할만 합니다.

그래서 리프에 `LinkedList` 로 연결되어 있는 특정 범위의 경우 상위로 가는 것이 아닌 옆으로 이동만 하면 됩니다.

B는 모든 노드에 데이터 저장, 상위 노드가 가이드이자 데이터 저장, 리프간 연결 안됨

B+ 는 리프에만 데이터 저장, 상위 노드는 그냥 인덱스 가이드 역할, 리프간 일렬로 연결되어 있다!
