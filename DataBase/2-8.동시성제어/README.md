### **Q. DB에서 동시성 제어를 어떻게 처리하나?**

데이터베이스는 크게 3가지 방식으로 동시성을 제어합니다.

첫 번째는 Locking 방식입니다. 읽기용 공유 락과 수정용 배타 락을 사용하여 트랜잭션 간 충돌을 방지합니다. 하지만, 락은 성능 저하나 데드락을 유발할 수 있다는 단점이 있습니다.

그래서 이를 보완하기 위해 락 없이 제어하는 낙관적 락과 MVCC 또한 제공합니다. 낙관적 락은 데이터에 버전이나 타임스탬프를 두어 수정 시점에 충돌을 확인하고, MVCC는 Undo Log 등에 저장된 데이터 스냅샷을 통해 읽기와 쓰기가 서로를 방해하지 않도록 일관된 읽기를 보장합니다.

</br>
</br>

**1. Locking (=비관적 Lock)**

트랜잭션이 데이터에 접근하기 전 Lock을 획득하여 다른 트랜잭션의 접근을 막는 방식

- 공유 락 (Shared Lock, S-Lock): 읽기 전용 락, 여러 트랜잭션이 동시에 읽을 수 있지만 수정은 불가능하다.
- 배타 락 (Exclusive Lock, X-Lock): 쓰기 전용 락, 락을 획득한 트랜잭션만 읽기/쓰기가 가능하며 다른 트랜잭션은 접근할 수 없다.
- 예시 1 : 공유락 VS 배타락
    
    ```
    
    T1 : 데이터에 대해 S-Lock을 요청하고 획득
    T2 : 같은 데이터에 대해 S-Lock 요청하고 획득 (읽기끼리는 허용되므로 즉시 획득)
    T3 : 같은 데이터를 수정하기 위해 X-Lock 요청
    -> T1와 T2가 Lock을 해제하기 전까지 T3는 대기 상태
    ```
    
- 예시 2 : 배타락 독점 상황 (누군가 수정 중이면 읽기 못 함)
    
    ```
    T1 : 데이터에 대해 X-Lock을 요청하고 획득 (데이터 수정 중..)
    T2 : 데이터 읽기 위해 S-Lock 요청 => T1이 수정하고 있어서 T2는 대기
    T3 : 데이터 수정을 위해 X-Lock 요청  => 역시 T1이 끝날 때까지 대기
    -> T1이 커밋되면 대기 순서에 따라 다음 트랜잭션이 락을 이어받음
    ```
    
- 한계: 락을 너무 많이 걸면 성능이 저하되고, 서로의 락이 풀리길 기다리는 데드락이 발생할 수 있다.

</br>

**2. 낙관적 격리 수준 제어**

트랜잭션 간 충돌이 적을 것이라고 '낙관적'으로 가정하는 방식

- 데이터를 읽을 때 락을 걸지 않고, 수정 시점에 내가 읽었던 데이터가 변했는지 버전(Version)이나 타임스탬프를 확인
    
    ```
    T1 : 데이터A 조회 (락 없음) -> v1 확인
    T2 : 데이터A 조회 (락 없음) -> v1 확인
    T1 : 데이터A 수정 후 커밋 -> DB의 현재 버전이 v1 이므로 수정 허용 + 버전 v2로 상승
    T2 : 데이터A 수정 -> T2는 v1로 알고 있는데, 실제 DB는 v2라서 불일치 + 업데이트 실패(rollback)
    T2 : 다시 데이터 조회 후 재시도
    ```
    
- 장점 : 락을 걸지 않으므로 성능상 이점이 큼
- 단점 : 충돌이 잦은 환경에서는 롤백 비용이 커서 비효율적임

</br>

**3. MVCC**

데이터의 스냅샷을 이용하여 동일한 데이터에 대해 여러 버전의 기록을 관리

→ 데이터를 읽는 트랜잭션과 쓰는 트랜잭션이 서로를 방해하지 않는다. 읽기 작업은 특정 시점의 버전(Undo Log 등)을 보게 되므로 락 없이도 일관성을 유지할 수 있다.

</br>

**4. 타임스탬프 순서 제어**

트랜잭션이 시작될 때 부여받은 ‘타임스탬프’를 기준으로 데이터 접근 순서를 제어하는 방식

→ 데이터마다 마지막으로 읽은 시간(read-ts)와 마지막으로 쓴 시간(write-ts)을 기록하고, 나중에 온 트랜잭션이 먼저 처리되었는지 확인

- 예시 : T2가 먼저 쓰고, T1이 나중에 쓰려 할 때
    
    ```
    T1 (10시 도착, TS: 100) : 데이터A를 조회 -> 데이터A의 R-ts가 100이 됨
    
    T2 (11시 도착, TS: 110) : 데이터A를 수정 후 커밋 -> 데이터A의 W-ts가 110이 됨
    
    T1 (TS: 100) : 이제야 데이터A를 수정(Write)하려고 시도
    
    -> DB가 확인해보니, 데이터A의 W-ts(110)가 현재 수정을 시도하는 T1의 TS(100)보다 큼
    
    -> '이미 다른 트랜잭션(T2)이 데이터를 수정했음'으로 판단 -> T1 업데이트 실패 및 rollback
    ```

</br>

**💡 동시성 제어 선택 기준**

| 기법 | 기준 | 예시 |
| --- | --- | --- |
| 비관적 Lock | 충돌이 빈번, 금전적 손실이 클 때 | 계좌 이체, 선착순 티켓 예매, 재고 관리 |
| 낙관적 Lock | 충돌이 가끔 일어날 것으로 예상되는 서비스 |  |
| MVCC | 일반적인 읽기 위주 서비스 |  |
