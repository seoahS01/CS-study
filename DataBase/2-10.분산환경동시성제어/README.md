### Q. 분산 환경에서는 동시성 제어를 어떻게 하나요?

분산 환경에서는 서버가 여러 대이므로 서버 간 상태를 공유할 수 있는 분산 락을 사용하여 제어합니다. 물론 DB 락도 가능하지만, 분산 환경에서 모든 서버가 DB 락에만 의존하면 커넥션 풀 고갈 사태가 발생할 수 있고, 특히 외부 API 호출이 섞인 긴 로직에서는 DB 성능이 심각하게 저하될 수 있습니다. 이러한 이유로 DB 입구에서 순서를 미리 정해주는 분산락이 필요합니다.

</br>

**분산락**

여러 대의 서버가 공통된 자원에 접근할 때, 데이터베이스 외부의 독립된 공간에 락을 두어 한 번에 하나의 서버만 자원을 사용하도록 제어하는 기술

</br>

**왜 필요할까?**

단일 DB 락만 사용하면, 수많은 서버가 동시에 락을 얻기 위해 대기하면서 DB 커넥션 풀이 고갈되거나, 외부 API 호출이 포함된 긴 트랜잭션 동안 DB 성능이 급격히 저하될 수 있다. 이를 방지하기 위해 DB 진입 전 단계에서 순서를 미리 정해주는 분산 락이 필요하다.

</br>

**구현 예시**

- Redis → 성능과 처리량이 중요한 경우 사용
    - Redisson 라이브러리를 주로 사용한다. 이는 락을 획득할 때까지 계속 요청을 보내는 '스핀 락' 방식이 아니라, Pub/Sub 방식을 사용하여 락이 해제될 때만 알림을 받아 재시도하므로 Redis에 주는 부하가 적다
- MySQL의 네임드 락 → 인프라 비용 절감이 우선인 경우 사용
    - 별도의 Redis 인프라를 구축하기 부담스러울 때 사용
    - GET_LOCK() 명령어를 통해 특정 문자열에 대해 잠금을 획득한다. 데이터 레코드를 직접 잠그는 것이 아니기 때문에 인덱스나 테이블 락과는 별개로 동작하며, 분산 환경에서 저렴하게 락 기능을 구현할 수 있다.
- ZooKeeper를 이용한 분산 락 → 절대적인 정합성과 장애 복구가 중요한 경우 사용
    - 강력한 일관성을 보장해야 하는 대규모 시스템에 사용
    - 임시 노드(Ephemeral Node)를 생성하여 락을 획득하고, 서버가 다운되면 노드가 자동으로 삭제되어 락이 해제된다. 구현은 복잡하지만 안정성이 매우 높다.
