### **Q. Locking 단위 설정에 대해 설명해주세요.**

로킹 단위는 데이터베이스가 한 번에 잠금 처리하는 범위를 의미합니다.

가장 세밀한 Row Lock은 특정 행만 잠가 동시성은 가장 높지만, 수많은 락 상태를 관리해야 하므로 메모리와 CPU 소모 같은 Locking 오버헤드가 큽니다.

반면, Table Lock은 락 관리 비용은 최소화되지만, 한 번 락이 걸리면 전체 테이블에 접근이 불가능해 대기 지연이 증가합니다.

락의 범위가 커질수록 관리는 쉽지만 동시성이 낮아지고, 작아질수록 동시성은 높아지고 관리 비용이 크다.

</br>
</br>

**Locking 단위**

- Row Lock : 특정 행 하나만 잠금 → 관리할 락이 많아져 메모리 사용량 증가
- Page Lock : 여러 행을 담고있는 물리적 단위인 페이지를 잠금
- Table Lock : 테이블 전체 잠금 → 구조변경(DDL)시에 사용

</br>

**Locking 오버헤드**

락을 걸고 푸는 과정 자체에서 다음과 같은 비용이 발생한다.

- 메모리 소비 (CPU/RAM) : DB는 현재 어떤 데이터에 어떤 락이 걸려 있는지, 누가 대기 중인지 관리하는 Lock Table을 메모리에 유지함 →  락의 단위가 세밀할수록 이 관리 비용 증가
- 지연 시간 (Latency) : 락을 획득하고 해제하는 과정에서 CPU 연산이 필요하며, 다른 트랜잭션이 락을 해제할 때까지 기다리는 대기 시간 발생
- Context Switching : 락을 얻지 못한 프로세스가 '대기 상태'로 들어갔다가, 락이 풀렸을 때 다시 '실행 상태'로 돌아오는 과정에서 OS 레벨의 자원 소모 발생
