### **Q. 트랜잭션의 ACID란??**

트랜잭션의 ACID는 데이터베이스에서 트랜잭션이 정확하고 안정적으로 동작하도록 보장하는 4가지 핵심 속성입니다.

Atomicity(원자성)은 트랜잭션의 모든 작업이 전부 성공하거나 전부 실패해야 하는 속성입니다.

Consistency(일관성)은 트랜잭션 수행 전후로 데이터가 DB의 제약 조건과 비즈니스 규칙을 항상 만족하는 상태를 유지하는 속성입니다.

Isolation(독립성)은 동시에 실행되는 트랜잭션들이 서로 영향을 주지 않도록 Lock이나 MVCC를 통해 독립적으로 동작하도록 보장하는 속성입니다.

Durability(지속성)은 commit된 트랜잭션의 결과가 Redo Log와 WAL을 통해 시스템 장애가 발생하더라도 유지되는 속성입니다.

</br>
</br>

1️⃣ **Atomicity (원자성)**

- 개념 : 트랜잭션의 연산은 DB에 전부 반영되거나(commit), 전혀 반영되지(rollback) 않아야 함
- 예시 : 계좌 이체 과정은 ‘출금’과 ‘입금’으로 나뉘어 실행되는데, 만약 입금 과정에서 문제가 발생하면 출금까지 포함한 전체 트랜잭션이 rollback되어 모든 작업이 없던 일이 되어야 함.
- 구현 : Undo Log를 통해 보장

</br>

2️⃣ **Consistency (일관성)**

- 개념 : 트랜잭션 수행 전후에 데이터는 항상 DB가 정의한 제약 조건과 비즈니스 규칙을 만족하는 유효한 상태를 유지해야 함
- 예시 : DB에 동일한 id 값을 가진 데이터를 두번 삽입하려고 하면, 데이터베이스는 PK 제약 조건에 위배되기 때문에 두번째 Insert를 허용하지 않음
- 구현 : PK, FK, UNIQUE, NOT NULL 같은 DB 제약 조건과 애플리케이션 로직을 통해 보장

</br>

3️⃣ **Isolation (독립성)**

- 개념 : 둘 이상의 트랜잭션이 동시에 실행될 때 서로 영향을 미치지 않고 독립적으로 동작해야 함
- 예시 : 두 사용자가 동시에 같은 콘서트 좌석을 예매하려고 할 때, 한 트랜잭션이 좌석을 먼저 점유하고 커밋한 이후에만 다른 트랜잭션이 그 결과를 반영해 처리하게 하여 중복 예매가 발생하지않도록 함
- 구현 : Lock이나 MVCC를 사용하고, 격리 수준에 따라 성능과 정합성을 trade-off 함

</br>

4️⃣ **Durability (지속성)**

- 개념 : 성공적으로 완료된 트랜잭션의 결과는 시스템 장애가 발생하더라도 영구적으로 반영되어야 함
- 예시 : 트랜잭션은 commit되었지만, 디스크에 실제 데이터가 flush되기 전에 DB 서버 크래시나 전원 장애가 발생해 데이터가 디스크에 반영되지 않은 상황에서도 Redo Log를 통해 해당 변경 내용을 재적용하여 데이터를 유지해야 함.
- 구현 : Redo Log와 WAL을 통해 보장
