### **Q. MySQL Repeatable Read에서는 Phantom Read가 발생하지 않는 이유는?**

MySQL은 Next Key Lock을 지원해서 Phantom Read를 방지합니다.

레코드 락과 갭 락을 결합해 데이터 사이의 빈 공간까지 잠금으로써, 다른 트랜잭션이 새로운 데이터를 끼워 넣는 것을 차단합니다.

</br>

**💡 Next-Key Lock**

MySQL InnoDB의 기본 격리 수준인 RepeatableRead에서 Phantom Read를 막기 위해 사용하는 락 방식이다.

- 구성 : 레코드 락(Record Lock) + 갭 락(Gap Lock)
- 특정 인덱스 레코드와 그 직전의 빈 공간을 하나의 구간으로 묶어서 잠금 → 새로운 데이터가 그 사이에 삽입되는 걸 막을 수 있음

</br>

**🚨 Gap Lock의 데드락 상황**

갭락의 목적은 오직 데이터 삽입을 막는 것

갭락끼리는 서로 충돌하지 않고 동시에 여러 트랜잭션을 가질 수 있음 → 여러 트랜잭션이 갭을 막는다고 DB 무결성에 문제가 없으니 전부 허용해줌

```
트랜잭션 A: id 10~20 사이에 갭 락을 획득합니다. (성공)

트랜잭션 B: 동일한 id 10~20 사이에 갭 락을 획득합니다. (성공, 갭 락은 중복 허용)

트랜잭션 A: 해당 범위에 id 15를 INSERT 시도합니다. → B의 갭 락이 해제될 때까지 대기합니다.

트랜잭션 B: 해당 범위에 id 16을 INSERT 시도합니다. → A의 갭 락이 해제될 때까지 대기합니다.
```

하지만 이렇게 서로 다른 트랜잭션이 서로 기다리는 데드락이 발생할 수 있음

→ 데드락 발생의 가장 빈번한 이유가 갭락 이라고 한다
