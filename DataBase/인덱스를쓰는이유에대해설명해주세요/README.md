# 인덱스를 쓰는 이유에 대해 설명해주세요.

### 먼저 인덱스를 쓰지 않는 예시를 보겠습니다.

| name | price | quantity |
| --- | --- | --- |
| 망고 | 20000 | 10 |
| 딸기 | 12000 | 20 |
| 수박 | 30000 | 5 |
| 사과 | 5000 | 30 |

위 처럼 과일 목록이 있습니다.

그런데 우리는 사과의 가격이 올라 `7,000원`으로 수정해야 합니다.

이에 따라 다음의 쿼리를 작성했습니다.

<br>
<br>

```sql
UPDATE FRUIT SET PRICE = 7000
WHERE NAME = "사과";
```

먼저 이름이 사과인 것을 찾아야 합니다.

일반적으로 가장 위에서부터 찾습니다.

망고 -> 딸기 -> 수박 -> 사과 ...

그런데 망고의 경우는 럭키하게 한 번에 찾았지만, 사과는 4번 만에 찾을 수 있었습니다. (Full Scan)

그런데 컬럼이 100개라면? 10000개라면? 1억개라면?

<br>
<br>
<br>

### 우리는 그래서 인덱스를 붙였습니다.

우리는 `사과`에 인덱스를 붙였습니다.

이제 인덱스 테이블에 다음과 같이 관리가 될 것 입니다.

(예시 입니다.)

| Index Key | (Name),Data Pointer (Address) |
| --- | --- |
| 딸기 | 0x002 (Block 1) |
| 망고 | 0x001 (Block 1)|
| 사과 | 0x004 (Block 2)|
| 수박 | 0x003 (Block 2)|

위는 가나다 순으로 **정렬**되어 있기에 사과의 `사`를 찾을 때는 딸기와 망고를 건너 뛰고 이진 탐색 등으로 빠르게 접근할 수 있습니다.

그러나 일반적으로 DB 크기의 약 10% 정도의 추가 공간이 필요하며 데이터 변경 작업이 자주 일어나면 오히려 성능에 영향이 생깁니다.

따라서 다음의 경우에 적용합니다.
1. 해당 컬럼이 `WHERE` 조건에 자주 사용되거나

- 이는 쿼리 실행 시 "내가 찾는 데이터가 어디있어?" 를 하는 필터링 역할입니다.

2. 해당 컬럼이 `JOIN` 조건에 자주 사용되거나

- 이는 두 개 이상의 테이블을 연결하는 역할입니다.
- 보통 한 쪽 테이블(Outer Table) 의 행 하나를 집어서 다른 쪽(Inner Table) 에서 일치하는 행을 찾아내는 과정을 반복합니다.
- 만약 연결되는 컬럼(FK)에 인덱스가 없다면, Outer Table 의 행 개수 만큼 Inner Table 전체를 반복해 찾아야 합니다.


3. 테이블의 변경이 자주 일어나지 않는 경우

## 한 줄 요약
> 인덱스를 사용하는 핵심 이유는 데이터 검색 속도의 향상을 위해서입니다.
> 데이터량이 커질수록 조건에 맞는 데이터를 찾기 위해 전체 테이블을 읽는 `Full Scan` 은 부하를 주게 됩니다.
> 이 때 인덱스를 활용해 B-tree 등 자료구조를 통해 탐색 범위를 줄여 쿼리 응답 속도를 단축하고 ThroughPut을 높일 수 있습니다.
> 그러나 인덱스는 추가 저장 공간이 필요학, `INSERT`, `UPDATE`, `DELETE`가 발생할 때마다 인덱스 구조 또한 갱신해야 하기에 쓰기 성능이 저하될 수 있습니다.
> 따라서 인덱스는 데이터 변경이 적고 조회가 빈번하며, WHERE 절이나 JOIN 조건에 자주 사용되는 컬럼에 적용하는 것이 효율적입니다.
